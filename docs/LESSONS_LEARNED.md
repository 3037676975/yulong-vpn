# 这次项目整理学到的经验

## 1. 版本必须统一

这次最大的问题是版本混乱。我们同时遇到过老插件、临时插件、不同检测版。以后必须只认一个母版：

```txt
v18-backend-check
```

后续插件升级必须基于这个母版继续做。

## 2. UI 和逻辑不要混

用户认可的是 v18-backend-check 的黑金界面。以后发新版时：

```txt
UI 不动
检测逻辑不动
只改后台地址、版本号、图标和必要文案
```

## 3. 后台地址要统一

所有地方统一使用正式后台域名：

```txt
https://api2.smilechat.cn
```

不要在不同文件里留下多个旧 Vercel 地址。

## 4. 账号密码设置要放在后台内部

正确方式：

```txt
登录后台 -> 左侧账号密码设置 -> 当前后台内弹出设置面板
```

不要做一个独立公开设置页。

## 5. 敏感配置只能放服务端

数据库服务端密钥、后台密码、验证码密钥都只能放在 Vercel 环境变量里。

不要放在：

```txt
插件源码
前端页面
公开文档真实值
截图
```

## 6. 正式主线必须聚焦

当前正式主线是：

```txt
Chrome 插件 + Web 后台 + Supabase 数据库
```

不属于当前主线的实验方向，不要放在 GitHub 首页展示，避免别人误解项目重点。

## 7. 文档很重要

开源项目不只是代码，还要让别人能看懂、能部署、能排查问题。

必须写清楚：

```txt
项目是什么
怎么部署
怎么配置数据库
怎么安装插件
怎么排查问题
哪些文件不要乱改
```

## 8. 发版前检查清单

每次发版前都检查：

```txt
1. 插件版本号是否正确
2. 插件后台地址是否正确
3. 插件 UI 是否还是 v18-backend-check
4. 后台是否能登录
5. 验证码是否能保存和验证
6. 节点是否能刷新
7. 后台检测是否能运行
8. README 是否同步更新
9. 是否误提交 .env
10. 是否混入旧版本压缩包或测试包
```

## 9. 后续建议

后续建议按这个顺序优化：

```txt
1. 把正式插件源码放入 extension/ 目录
2. 增加一键打包插件脚本
3. 增加 GitHub Actions 自动 npm build
4. 后台页面拆分组件
5. 替换开源仓库里的真实业务节点为演示节点
```
